Zaizai Isle
首页渐进更新 PRD（AI 执行版）
文档性质：执行约束文档（Execution Guard）使用对象：AI（设计 / 文案 / 前端 / Agent）目标：在不破坏现有页面的前提下，逐步校准叙事与模块职责

1. 项目背景（Context）
Zaizai Isle 是一座可持续生长的个人数字小岛，不是传统意义上的：
* 个人主页
* 作品集
* 产品官网
* 信息流平台
当前版本已经上线并可用，本 PRD 不允许推翻现有结构，只允许在其之上进行渐进式校准与增强。

2. 本轮更新的核心目标（Objectives）
主目标（Only One）
防止 Zaizai Isle 被误读为“作品集 / 简历 / 产品列表”
次目标
* 明确页面的“阅读顺序”
* 让模块各司其职，不争夺叙事
* 强化“正在生长”的感知，而非“已完成”

3. 非目标（Explicit Non-Goals）
本轮更新 不做 以下事情：
* ❌ 不更换整体视觉风格
* ❌ 不大改布局结构
* ❌ 不引入新的页面层级
* ❌ 不重写全部文案
* ❌ 不引入复杂交互或动画系统

4. 页面叙事分层规范（Hard Constraint）
页面同时存在三层叙事，但同一时间只允许一层主导。
4.1 运行层（System Layer）
* 性质：基础设施
* 表达方式：状态、事实、边界
* 可见性：低
* 语言：冷静、陈述式
4.2 世界层（Island Layer）【当前主导】
* 性质：空间与体验
* 表达方式：区域、节奏、环境
* 可见性：高
* 语言：克制、具象、不解释
4.3 意义层（Meaning Layer）
* 性质：方向锚点
* 表达方式：极简短句
* 可见性：极低（全站 ≤ 3 处）
* 禁止展开说明

5. 页面整体结构（冻结项）
5.1 页面形态
* 单页结构（One-page）
* 模块纵向排列
* 无强制阅读路径
5.2 模块清单（不可删除）
* Identity Card
* Core Build Card
* Weather Card
* Stats Card
* Works Card
* Tools Card
* Archive / Guestbook
* Footer / Boundary

6. 模块级更新策略（重点）
6.1 Core Build Card（最高优先级）
模块职责：
* 作为“构建事实锚点”
* 回答：这座岛现在正在被建造成什么
必须包含的元素：
1. 非量化构建进度视觉（进度条 / 状态条）
2. 模块运行状态列表（active / forming）
3. 一句低调的构建声明（非口号）
禁止：
* 插画 / IP
* CTA
* 情绪化语言

6.2 Identity Card
调整方向：
* 从“我是谁” → “你现在在谁的岛上”
* 保留 IP（鲸头鹳）作为气质锚点
* 不承担系统解释职责

6.3 Works Card
阅读方式重定义：
* 从“作品集” → “已建成结构”
* 减少数量，强调系统性
* 不强调视觉截图本身

6.4 Stats Card
语义约束：
* 表示“发生过什么”
* 不表示成就
* 不用于对比或炫耀

6.5 Weather / Environment
定位：
* 世界层氛围模块
* 后置，不抢主叙事
* 不承担说明功能

6.6 Archive / Guestbook
重定义：
* 从“互动” → “痕迹 / 时间沉积”
* 弱化社交感
* 不引导频繁参与

7. 执行顺序（AI 必须遵守）
1. 只改模块“第一感知层”
    * 标题
    * 第一行说明
    * 状态提示
2. 不动模块主体内容
3. 一次只改一个模块
4. 每轮修改后进行误读检查

8. 误读检查清单（Self-Check）
每次输出前，AI 必须自检：
* 是否会被读成“作品集”？
* 是否在解释自己？
* 是否出现过多情绪或诗性语言？
* 是否抢夺了世界层的主导权？
如任一为“是”，则修改失败，需回退。

9. 成功标准（Success Criteria）
* 用户快速理解：这是一个正在生长的地方
* 即使不读文字，也能感知“未完成性”
* 页面可以长期扩展而不失去一致性

10. 使用方式（给未来的你 / AI）
当你想“再加点什么”的时候，请先问一句：这是在扩建岛，还是在装饰页面？
如果是后者，请暂停。

11. V1.2.6 结构精益化与性能校准（2026-02-09）
在 V1.2.6 的持续完善中，对页面进行了深度性能审计与结构精度升级：
* **12 列高精度栅格**：将原有的 6 列栅格升级为 12 列，提升了卡片宽度控制的分辨率。
* **物理比例锁定**：将基础行高（auto-rows）从 192px 映射至 220px，配合 `screen-2xl` (1536px) 的全屏宽度，保持了最小卡片（Stats/Social）的 1:1 正方形比例。
* **构建面板（Core Build）密度重构**：
    * 采用双列并排显示 8 个核心模块，大幅提升视觉信息密度。
    * 进度条升级为 32 单元高分辨率模式，加入 System Scanner 动态效果。
* **i18n 全量对齐**：所有模块名称及状态值均已托管至 `language-context.tsx`。
在完成 12 列栅格升级后，引入了 Vercel React Best Practices 进行深度性能审计：
* **动态模块加载**：核心 Bento 组件（如 Weather/AI Hub）采用 `next/dynamic` 异步加载，显著减小了首屏 JS 负载。
* **渲染拓扑优化**：对 `IdentityAnchor` (IdentityCard) 内部的装饰性元素（云朵/动画）进行了 `React.memo` 隔离，规避了高频帧动画引起的父组件脏检查开销。
* **静态提升**：将所有栅格配置（Span Mappings）及纯函数提升至组件作用域外，减少了渲染时的对象创建。
* **状态精简**：移除冗余的 `date` 副作用计数，保持系统主循环的纯净度。

12. V1.2.6 文档系统与视觉减法（2026-02-10）
在性能底座稳固后，针对岛屿的“叙事交互”与“视觉噪音”进行了二次精调：
* **文档系统 2.0 (Documentation Overhaul)**：
    * 推出 V1.2.6 版主说明书 `PRODUCT_DOCUMENTATION.md`，修复了 Emoji 渲染偏移。
    * 自研 `TOCNav` 客户端组件，利用 `getBoundingClientRect` 模拟局部滚动容器锚点逻辑，解决了嵌套布局下的导航失效问题。
* **环境感知交互 (Vibe Context)**：
    * 引入 URL Anchor 亮度暗示机制 (`#dark-bg` / `#light-bg`)。
    * 实现了 UI 文字色值随背景图片的明暗程度自动进行语义化对齐。
* **视觉噪音精简 (System Sanitization)**：
    * 简化 `SystemStatus` (顶栏) 和 `SystemBoundary` (页脚)，移除冗余指标，强化“数字建筑”的安静气质。
* **构筑面板密度修正**：
    * 优化 `CoreBuildCard` 样式，提升 12 列布局下的视觉一致性。
